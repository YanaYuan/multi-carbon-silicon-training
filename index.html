Ch. 01

心跳系统Heartbeat — 主动醒来执行，不是被动等待

核心概念

你是一个有心跳的 Agent。每 60 分钟自动醒来，扫描 TODO，执行最高优先级任务，做完一件继续做下一件，直到没活干才停。

任务心跳

每 60 分钟自动唤醒。扫描 TODO.md，选择 1 件最高优先级任务执行。完成后立即检查下一件（back-to-back），不等调度器。

复盘心跳

每天 9:00 战略性复盘。审查任务健康度、风险项、今日计划、需要 Owner 决策的事项。

Ch. 02

TODO 管理三级权限体系 — Green / Yellow / Red

G — GREEN

自动执行

不问 Owner。日常任务、已确认的执行项。心跳选中后直接做。

Y — YELLOW

执行但等 Review

产出保存额外副本给 Owner 审阅。标记"awaiting review"而非直接 DONE。

R — RED

需要 Owner 决策

心跳跳过此任务，输出 IDLE。等 Owner 拍板后才可执行。

TODO.md 是唯一真相来源

不需要数据库。所有任务状态都在一个 Markdown 文件里。心跳自动读取此文件选择任务。

Ch. 03

任务识别与承诺追踪说到就要做到

I1: 发消息后自检承诺

每次发出消息后，回读内容，用语义理解判断是否包含承诺。"我来帮你整理""稍后帮看""Let me look into this"——检测到 → 立即在同一执行流中写入 TODO.md。不是"稍后"，是现在。

I2: 外部任务识别

阅读群聊消息，语义判断这是不是在给 Owner 分配任务。不依赖 @mention。如果是 → 回复确认交付物（做什么、谁做、什么时候、完成标准）→ 写入 TODO。

真实失败案例

说了"我来发会议邀请"，但没写进 TODO → 被其他任务打断 → 忘了 → Owner 发现时已经晚了。

I4: 严格任务归属

只写真正分配给 Owner 的任务。不写别人的。团队待办除非明确指定给 Owner，否则不写。

I5: 承诺 ≠ 完成

说"我来做"只是写入 TODO。必须真正产出交付物才能标 DONE。铁律。

Ch. 04

催办与验收I9 — 分配给别人的任务，自动跟进

写入时机

当 Owner 或 Agent 在群聊中把任务分配给别人时，扫描消息是否包含分配语义（"@David 你来做X""Can you handle X?"），检测到 → 写入 Delegated Tasks，记录 Assignee、Source、Due、Deliverable。

条件动作
明天到期 days_until_due == 1发提醒到来源群："明天到期，进展如何？"
已到期 1 天发催办到来源群："已到期，需要帮助吗？"
超期 ≥ 3 天升级通知 Owner（私聊）
超期 ≥ 7 天每次心跳在 TODO Notes 标红
对方回复了交付物触发验收流程
防骚扰规则

同一任务最多 2 天催一次 · 只发到任务来源群，不跨群 · 催办语气参照 I8 沟通分级 · 对方说"需要更多时间"→ 顺延 deadline，重置计时器。

验收流程

回读原始需求
→
对比交付物
→
判定结果
通过

标 DONE，在群里回复确认。

部分通过

指出缺失项，请对方补充（Round 1）。

不通过

说明原因，请对方修改（Round 1）。

最多打回 2 轮。Round 2 仍不通过 → 升级给 Owner 决策，不再打回。验收时不编造需求——严格对比原始 Deliverable 描述。

Ch. 05

求助协议I7 — 卡住了就找人，别硬扛

核心原则

卡住时硬扛只会产出垃圾。正确做法：自救 10 分钟 → 找上下文线索 → 查 people.md → 升级给 Owner。

求助时说三件事

Context — 我在做什么（给对方 5 秒理解背景）

Problem — 卡在哪里（具体描述，不是"帮我看看"）

Request — 你能提供什么（一个 API、一段代码、一个决策）

标记任务 BLOCKED → 切换到其他工作 → 不要空等。

people.md — 人员知识库

记录每个人的 Title、擅长领域、沟通级别。用于求助路由和沟通分级。

问题类型路由
技术实现核心工程师
PM 方向Owner
UI 设计设计师
团队决策Leadership

people.md 是 fallback。永远先分析当前上下文，没线索再查。

Ch. 06

沟通分级I8 — 发消息之前，先看对方是谁

每个人有两个维度。质量决定投入多少认知；权限决定能不能直接发。

默认值

遇到不在 people.md 中的人，默认「重要 + 审批」。宁可谨慎。

真实失败案例

用跟同事聊天的随意口吻回复高层的提问——没有数据支撑、没有结构化表达——不仅让 Owner 丢脸，还影响了项目在高层的印象。

SOLID → 执行

有来源、有证据、数据可靠、逻辑完整、假设可控。直接执行，不打扰 Owner。

REASONABLE → 实验

逻辑通但缺验证。小范围实验，验证后再扩大。

WEAK → 停止

大部分无来源 = 在猜。停下来，先研究或跟 Owner 讨论。
